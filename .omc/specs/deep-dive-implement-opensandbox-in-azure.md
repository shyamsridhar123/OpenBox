## Goal

Implement Alibaba's OpenSandbox — a general-purpose AI-agent sandbox platform (FastAPI control plane, an execution daemon, Kubernetes runtime with CRDs, gVisor/Kata/Firecracker isolation, multi-language SDKs) — on Microsoft Azure, with **conceptual parity to Azure Dev Box's governance and image-management model** (DevCenter→Project→Pool hierarchy, image-definition YAML, RBAC, network connections, autostop).

**v1 (4-6 weeks):** A working agent-callable sandbox service in East US 2 supporting ~500 concurrent active sandboxes, with per-user Entra-authenticated REST API + Python/JS/Go SDKs + signed curated base images + a read-only observability portal + Notation/Ratify signature enforcement + Azure Policy in Deny mode.

**Beyond v1 (deferred):** Full self-service human portal (write operations), multi-region, image-definition YAML pipeline, all 5 SDKs (add Java + C#), TTL idle reaper, hibernation/CRIU, Windows sandboxes.

## Constraints

### Architecture
- **Compute substrate:** Hybrid — **Azure Container Apps (ACA)** hosts the FastAPI control plane + read-only portal frontend + portal API; **AKS** hosts OpenSandbox's K8s controller + CRDs + execution-daemon DaemonSet + sandbox pods.
- **Sandbox isolation:** AKS **Pod Sandboxing with Kata Containers** (GA) on Azure Linux 3.0 + Gen2 VMs (`runtimeClassName: kata-vm-isolation`) is mandatory for all sandbox pods. Non-sandbox workloads (controller, daemon) use `runc`.
- **Tenancy model:** Agent-as-tenant — single AKS cluster, namespace-per-user (or session-scoped pods within a user namespace). Kata provides the primary trust boundary; namespaces provide soft logical separation.
- **Region:** East US 2 (single region for v1; architecture must be region-parameterizable).
- **Cross-plane auth:** ACA → AKS via signed JWT (Entra-issued, validated against JWKS).

### Identity / Auth
- **User → OpenSandbox API:** Per-user **Microsoft Entra tokens via OBO (On-Behalf-Of) flow**. FastAPI validates JWT via Entra JWKS endpoint.
- **Sandbox → Azure resources:** **Entra Workload Identity** with federated credentials. One **User-Assigned Managed Identity (UAMI) per user namespace** (respecting the 20-federated-credentials-per-UAMI ceiling).
- **Service-principal fallback:** Out of scope for v1 (per-user only).
- **Portal users:** Entra ID SSO via ACA Easy Auth; RBAC at Azure resource level (custom roles mirror Dev Box: `SandboxCenter Admin` / `SandboxProject Admin` / `Sandbox User`).

### Network
- **AKS cluster:** Private (API server on private endpoint).
- **Egress:** NAT Gateway per subnet for SNAT; all sandbox egress routed via UDR to **Azure Firewall Premium** with explicit allowlist (PyPI, npm, pip mirrors, configurable). OpenSandbox's per-sandbox egress controls are an additional in-cluster enforcement layer.
- **In-cluster:** **Azure CNI Powered by Cilium** + **Advanced Container Networking Services (ACNS)** for L3-L7 NetworkPolicy and FQDN filtering at pod level.
- **ACR / Key Vault:** Private endpoints only; no public access.

### Images
- **Registry:** **ACR Premium** (zone-redundant; geo-replication enabled but only 1 region active for v1).
- **Base images:** Curated catalog of 3-4 base images for v1 (`base/python:3.12`, `base/node:20`, `base/go:1.22`, optional `base/datascience`). Maintained by platform team.
- **Customization model:** Dev-Box-style — **YAML image-definition** layered on a curated base, baked via **ACR Tasks** into reusable images. (v1 ships the curated bases; the YAML pipeline ships in v1.5.)
- **Signing:** All images signed with **Notation (Notary v2)** using an Azure Key Vault-backed signing cert. Signatures stored as OCI referrers in ACR.
- **Admission enforcement:** **Ratify + Azure Policy Gatekeeper constraint** denies pods whose image lacks a valid signature.

### Governance
- **Azure Policy add-on for AKS** enabled. **K8s pod security restricted** built-in initiative assigned in **Deny mode**. Custom denies: privileged containers, hostPath volumes, non-signed images, non-ACR registries, `runAsNonRoot=false`.
- **Defender for Containers** enabled on registry + cluster. (Known gap: Defender does **not** assess Kata-runtime pods — accept and document.)
- **ResourceQuota** + **LimitRange** per user namespace.

### Observability
- **Container Insights** → **Log Analytics workspace** (one workspace for v1).
- **App Insights** instruments the FastAPI control plane + portal API (OpenTelemetry).
- **Cilium flow logs** + **AKS audit logs** + **Azure Firewall logs** → same Log Analytics workspace.
- **Sandbox audit log:** daemon stdout/stderr → Fluent Bit → Log Analytics with structured `user_id`, `session_id`, `command`, `exit_code` fields.

### Scale
- **v1 target:** ~500 concurrent active sandboxes.
- **AKS sizing:** Kata node pool of 3-10 nodes (Standard_D8s_v5, Gen2, Azure Linux 3.0) with cluster autoscaler enabled. Separate `runc` system node pool for controllers/daemon/CoreDNS.
- **ACA:** Min 1 replica per service, scale to 10 via KEDA HTTP scaler.

### IaC
- **All Azure resources:** **Bicep modules** organized by service (aks.bicep, aca.bicep, acr.bicep, kv.bicep, entra.bicep, network.bicep). One root `main.bicep` with parameter file per environment.
- **App deployments:** Helm charts for AKS workloads; ACA deployment via `az containerapp` + Bicep `containerApps` resource.

## Non-Goals (v1)

- ❌ Windows sandboxes (Linux-only).
- ❌ GUI / RDP / desktop access into sandboxes.
- ❌ Multi-cloud or hybrid on-premises deployment.
- ❌ Multi-region (architecture must support it; v1 deploys to a single region).
- ❌ Hibernation / CRIU state preservation (sandboxes are stateless for v1).
- ❌ External-customer multi-tenancy (agent-as-tenant model assumes single trust domain).
- ❌ Service-principal-only auth (per-user Entra OBO required).
- ❌ Full self-service write portal (read-only for v1; full portal in v1.5).
- ❌ Java and C# SDKs (Python/JS/Go for v1; add later).
- ❌ Visual-Studio-style marketplace images.
- ❌ Intune enrollment, Entra hybrid join, Conditional Access compliant-device policies (no container equivalent).
- ❌ TTL idle reaper (sandboxes destroyed by explicit API call for v1; auto-reap in v1.5).
- ❌ ACI Confidential Containers / Confidential AKS (SEV-SNP). Architecture allows future addition.

## Acceptance Criteria

### A. Architecture deployed end-to-end
1. `az deployment sub create` executes `main.bicep` and provisions: 1 resource group, 1 AKS cluster (private, Cilium-ACNS, 1 system pool + 1 Kata pool), 1 ACA environment, 1 ACR Premium, 1 Key Vault (Notation cert), 1 Log Analytics workspace, 1 App Insights, 1 Azure Firewall Premium, 1 NAT Gateway, 1 VNet (≥3 subnets), Entra app registration + UAMI(s), all RBAC role assignments.
2. AKS Pod Sandboxing (Kata) verified: `kubectl get runtimeclass` shows `kata-vm-isolation`; sample pod with `runtimeClassName: kata-vm-isolation` reaches `Running`.
3. ACA control plane responds to `GET /healthz` with 200 over private FQDN (private endpoint or internal ingress).
4. Cross-plane auth: ACA pod successfully calls AKS-hosted controller with a signed JWT; non-signed request returns 401.

### B. Sandbox lifecycle via REST API
5. `POST /sessions` (with valid user Entra token) creates a sandbox pod in `ns-<user-oid>` namespace, returns `session_id` and connection details. Pod uses `runtimeClassName: kata-vm-isolation` and an ACR-hosted, Notation-signed base image.
6. `POST /sessions/<id>/run` runs a shell command in the sandbox; returns stdout/stderr/exit_code. End-to-end latency from request to first byte of stdout ≤ **3 seconds** (cold) / ≤ **500 ms** (warm pool).
7. `DELETE /sessions/<id>` terminates the pod and cleans up associated PVCs / network policies.
8. Concurrent-load test: 100 simultaneous `POST /sessions` requests succeed within p95 < 10 s.

### C. Identity end-to-end
9. Calling `/sessions` without an Entra token returns 401.
10. Calling with a token for User A but trying to read User B's session returns 403.
11. Inside the sandbox, an `az` CLI call (using the projected service-account token) successfully reads a Key Vault secret scoped to User A's UAMI; the same call for a secret scoped to User B's UAMI returns 403.
12. Audit log query in Log Analytics returns the full chain (`user_oid → session_id → command → exit_code → egress_dest`) for any session_id within 60 s.

### D. Image governance
13. Pushing an unsigned image to ACR and trying to schedule a pod from it is rejected by Ratify+Gatekeeper with a clear admission-error message.
14. Pushing a signed image (Notation signature in ACR) succeeds and schedules correctly.
15. Defender for Containers shows CVE scan results for all 3-4 curated base images.
16. Attempting to schedule a pod from a registry other than the configured ACR is denied by Azure Policy.

### E. Network isolation
17. Sandbox pod cannot egress to an arbitrary internet IP not on the firewall allowlist (verified by `curl` to a non-allowed destination → blocked).
18. Sandbox pod cannot reach another user's namespace pods (verified by cross-namespace HTTP attempt → blocked by Cilium NetworkPolicy).
19. ACR / Key Vault DNS resolves to private endpoint IPs from within the cluster.

### F. SDKs
20. Python SDK: `pip install opensandbox-azure` then `client.create_session()` → `client.run("echo hello")` returns `"hello\n"`.
21. JS SDK: equivalent flow in Node.js.
22. Go SDK: equivalent flow.
23. All three SDKs handle Entra token acquisition via MSAL or equivalent (no hardcoded credentials).

### G. Observability portal
24. Portal at `https://<aca-fqdn>/` loads behind ACA Easy Auth (Entra SSO).
25. Portal lists active sessions for the logged-in user with `session_id`, `image`, `created_at`, `state`.
26. Portal shows audit log entries for the user's sessions (last 24 h, filterable by `session_id`).
27. Portal shows quota usage (current vs. limit) for the user's namespace.
28. Portal **does not** expose any write/destroy operation in v1 (those go through the REST API only).

### H. Documentation & runbooks
29. `README.md` covers: architecture diagram, quickstart, IaC parameters, SDK usage examples.
30. `runbooks/incident-response.md` covers: how to revoke a user's access, how to quarantine a compromised pod, how to rotate the Notation signing cert.
31. `docs/threat-model.md` documents the agent-as-tenant trust model, Kata's role, known gaps (Defender + Kata).

### I. CI/CD
32. PR triggers: Bicep `what-if` + ARM-TTK + Helm lint + Python/JS/Go SDK unit tests.
33. Main-branch merge: image build (ACR Tasks) → Notation sign → push → AKS rollout (Helm) + ACA revision update.
34. Image build pipeline rejects images with high/critical CVEs from Defender scan.

## Assumptions Exposed

1. **OpenSandbox upstream uses cluster-scoped CRDs and a controller.** If source inspection (probe in trace) reveals it only uses standard Pod API + `runtimeClassName`, the architecture could simplify to ACA-only — but spec assumes CRDs as per README.
2. **OpenSandbox upstream is forkable / modifiable.** Some Entra integration (token validation in FastAPI) is not upstream; we accept maintaining a thin fork or contributing patches.
3. **User identity is a Microsoft Entra ID user object**, not a non-Entra identity provider. (B2B/external users via Entra B2B guest accounts are acceptable; non-Entra IdPs are out of scope.)
4. **Azure subscription has the AKS Pod Sandboxing feature enabled and Defender for Containers licensed.**
5. **Latency target of ≤3 s cold start** assumes Kata cold start measured on Standard_D8s_v5 + image already cached on node. Worst case (uncached image) may be 10-15 s; if unacceptable, a warm-pool pre-pull strategy is required.
6. **OpenSandbox's `api_key` auth is replaced or layered**, not used as-is — production identity is Entra OBO.
7. **CRIU / hibernation is explicitly deferred to v1.5+**, accepting stateless-only sandboxes for v1.
8. **A 4-6 week build assumes a small team (2-3 engineers) with prior AKS + Bicep + FastAPI experience.** Less experienced team: 8-10 weeks.

## Ontology (Final)

| Term | Definition |
|---|---|
| **Sandbox** | An ephemeral Kata-isolated Linux pod scheduled in a user's namespace, running an ACR-signed base image, with per-pod egress policy. Lifecycle: created via REST, destroyed via REST. Stateless in v1. |
| **Session** | A unit of sandbox usage: `session_id = uuid`, owned by exactly one user OID, mapped 1:1 to a pod for v1. |
| **User** | A Microsoft Entra ID identity (member or B2B guest). Authenticates via OBO; owns a namespace `ns-<user-oid>` + UAMI. |
| **Project** | A Dev-Box-parity concept; for v1, maps to the user namespace. Will become a multi-user grouping in v1.5. |
| **Pool** | The set of (Kata node pool + curated image set) available to sandboxes. v1 has one pool. |
| **Image definition** | YAML spec layering custom tasks on a curated base. v1.5 feature. |
| **Curated base image** | A Notation-signed ACR image maintained by the platform team. v1 has 3-4. |
| **Control plane** | FastAPI + portal API + portal frontend on ACA. |
| **Runtime plane** | OpenSandbox controller + execution daemon + sandbox pods on AKS. |
| **Audit log** | Structured Log Analytics entry: `user_oid, session_id, command, exit_code, egress_dest, ts`. |
| **Notation signature** | Notary v2 signature stored as OCI referrer in ACR; enforced at admission by Ratify+Gatekeeper. |

### Ontology Convergence
Entities stabilized over the 5-question interview. No renames in the last 2 rounds. "Project" is the only term with intentional ambiguity (single-user in v1 → multi-user in v1.5), explicitly documented.

## Trace Findings

Full trace report: `.omc/specs/deep-dive-trace-implement-opensandbox-in-azure.md`.

### Most likely explanation (validated)
AKS + Kata as runtime substrate; ACA for control plane + portal; ~65% conceptual Dev Box parity (Windows-bound 35% formally dropped). Confirmed by interview answers.

### Per-lane critical unknowns — resolutions
- **Lane 1 (CRD scope):** Spec **assumes** OpenSandbox uses CRDs (per README). A discriminating probe — reading `github.com/alibaba/OpenSandbox/tree/main/kubernetes` — should run as **task #0** of the planning phase to validate the assumption before Bicep work begins.
- **Lane 2 (consumer surface):** Resolved — agents primary, humans get a read-only portal in v1, full portal in v1.5.
- **Lane 3 (tenant trust):** Resolved — agent-as-tenant model. Single trust domain, single cluster, Kata as primary boundary, namespaces as logical separator.

### Evidence that shaped the spec
- Lane 1's discovery that **AKS+Kata is GA** (not preview) and one-line activation via `runtimeClassName` made AKS the obvious primary substrate.
- Lane 2's parity table directly mapped 13 Dev Box features to OpenSandbox-on-Azure equivalents — drove the "Dev-Box→OpenSandbox-on-Azure mapping" section.
- Lane 3's warning that **Defender does not assess Kata pods** is captured as an assumption + known gap in the threat model.
- Lane 3's **20-federated-credentials-per-UAMI ceiling** drove the "one UAMI per namespace" decision (not per-session).
- Lane 1's identification of **ACA control plane + AKS runtime** as a strong #2 architecture was validated by the user choosing it directly in the interview.

### Recommended probe (still required at planning phase)
Read OpenSandbox's `kubernetes/` directory before locking down AKS RBAC. If the controller requires `cluster-admin` permissions or installs unexpected CRDs, the namespace-scoped RBAC assumed above must be widened.
